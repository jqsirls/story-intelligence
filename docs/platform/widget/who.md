# Storytailor Embeddable Widget - Who

**Last Updated**: 2025-12-17

## Team Ownership

The Storytailor Embeddable Widget is a critical component of the Storytailor platform, with primary ownership residing within the **Frontend Platform Team / Partner Integration Team**.

### Primary Owner

- **Team**: Frontend Platform Team / Partner Integration Team
- **Lead**: [Lead Engineer Name/Alias]
- **Contact**: #frontend-platform-team on Slack

### Secondary Contacts / Contributors

- **Team**: Partner Integration Team (for partner requirements and integration support)
- **Team**: Platform Infrastructure Team (for CDN configuration and deployment)
- **Team**: Product Management (for feature roadmap and widget priorities)
- **Team**: Design Team (for theming, UI/UX, and design system integration)

## Key Stakeholders

- **Product Management**: Defines widget feature priorities, partner integration roadmap, and customization requirements.
- **Engineering Team**: Responsible for widget implementation, theming system, and integration with Storytailor API.
- **Partner Integration Team**: Works with partners to ensure successful widget integration and customization.
- **Design Team**: Responsible for widget UI/UX, theming system, and design token management.
- **Platform Infrastructure Team**: Ensures widget CDN configuration, deployment, and performance optimization.
- **Marketing Team**: Leverages widget capabilities for partner-focused messaging and quick integration benefits.

## On-Call / Support

- **Primary On-Call**: Frontend Platform Team
- **Escalation**: Partner Integration Team
- **Support Channel**: #engineering-support on Slack

## Contribution Guidelines

- All code changes must be reviewed and approved by at least one member of the Frontend Platform Team.
- Theming and UI changes must be reviewed by the Design Team.
- Significant architectural changes (e.g., new widget features, API changes) require an RFC (Request for Comments) process involving relevant stakeholders.
- Widget implementations must maintain backward compatibility and support multiple website platforms (Webflow, Framer, Wix, custom).
- Documentation updates are highly encouraged and should follow the established [Naming Conventions](../../NAMING.md) and documentation guidelines.
- Security reviews are required for any changes affecting authentication, data handling, or privacy settings.
