# Storytailor SDKs - Who

**Last Updated**: 2025-12-17

## Team Ownership

Storytailor SDKs are critical components of the Storytailor platform, with primary ownership residing within the **SDK Development Team / Mobile Platform Team**.

### Primary Owner

- **Team**: SDK Development Team / Mobile Platform Team
- **Lead**: [Lead Engineer Name/Alias]
- **Contact**: #sdk-development-team on Slack

### Secondary Contacts / Contributors

- **Team**: Web Platform Team (for Web SDK and React Native SDK)
- **Team**: Mobile Platform Team (for iOS and Android SDKs)
- **Team**: Platform Integration Team (for API integration and backend coordination)
- **Team**: Product Management (for feature roadmap and SDK priorities)
- **Team**: Design Team (for UI/UX guidelines and design system integration)

## Key Stakeholders

- **Product Management**: Defines SDK feature priorities, platform support roadmap, and integration requirements.
- **Engineering Team**: Responsible for SDK implementation, API integration, and platform-specific optimizations.
- **Mobile Developers**: Primary users of iOS, Android, and React Native SDKs - provide feedback and requirements.
- **Web Developers**: Primary users of Web SDK - provide feedback and requirements.
- **Platform Infrastructure Team**: Ensures backend APIs and services support SDK requirements.
- **Marketing Team**: Leverages SDK capabilities for developer-focused messaging and positioning.

## On-Call / Support

- **Primary On-Call**: SDK Development Team
- **Escalation**: Platform Integration Team
- **Support Channel**: #engineering-support on Slack

## Contribution Guidelines

- All code changes must be reviewed and approved by at least one member of the SDK Development Team.
- Platform-specific changes (iOS, Android, Web, React Native) require review by the respective platform team.
- Significant architectural changes (e.g., new SDK features, API changes) require an RFC (Request for Comments) process involving relevant stakeholders.
- SDK implementations must maintain consistency across platforms while respecting platform-specific best practices.
- Documentation updates are highly encouraged and should follow the established [Naming Conventions](../../NAMING.md) and documentation guidelines.
- Breaking changes require deprecation notices and migration guides.
